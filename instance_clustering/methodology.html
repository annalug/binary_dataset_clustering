Uma abordagem baseada na distribuição das instâncias individuais para medir similaridade entre datasets heterogêneos de detecção de malware Android, sem depender de alinhamento de features.

Python · scikit-learn MMD · Wasserstein t-SNE · UMAP 11 datasets
// 02 — Metodologia

Pipeline de análise

O desafio central é que os datasets possuem espaços de features incompatíveis — alguns usam permissões Android brutas, outros usam métricas de grafo de API calls, outros usam features estatísticas de tráfego. A abordagem contorna isso operando no nível das instâncias, não das features.

1

Carregamento e limpeza

Cada CSV é carregado independentemente. Colunas constantes (variância zero) são removidas pois não carregam informação discriminativa. A coluna class é separada e não entra na análise de distribuição.

2

PCA independente por dataset → espaço latente comum

Cada dataset é projetado para um espaço latente de 20 dimensões via PCA, independentemente dos outros. Isso resolve o problema de features incompatíveis: não tentamos alinhar colunas, mas capturamos os padrões de covariância das instâncias de cada dataset.

# Para cada dataset individualmente: pca = PCA(n_components=20, random_state=42) X_latent = pca.fit_transform(X_original) # X_original: (n_samples, n_features_orig) ex: (1000, 4273) # X_latent: (n_samples, 20) ← espaço comparável

A intuição é que dois datasets com distribuições similares terão estruturas de covariância similares, mesmo com nomes de features diferentes.

3

MMD — Maximum Mean Discrepancy

Para cada par de datasets, calculamos o MMD entre as distribuições das instâncias no espaço latente. O MMD mede o quanto duas distribuições diferem usando um kernel RBF — retorna 0 quando são idênticas.

def mmd(X, Y, gamma=1.0): XX = rbf_kernel(X, X, gamma) # similaridade interna A YY = rbf_kernel(Y, Y, gamma) # similaridade interna B XY = rbf_kernel(X, Y, gamma) # similaridade cruzada return XX.mean() + YY.mean() - 2 * XY.mean() # Resultado: matriz D de shape (11, 11) com todas as distâncias
4

Clustering hierárquico sobre a matriz MMD

A matriz de distâncias MMD 11×11 é usada diretamente pelo AgglomerativeClustering com linkage average. O número de clusters é escolhido automaticamente testando k=2..8 e maximizando o Silhouette Score.

for k in range(2, 9): labels = AgglomerativeClustering( n_clusters=k, metric='precomputed', linkage='average' ).fit_predict(D_mmd) score = silhouette_score(D_mmd, labels, metric='precomputed') # Melhor k = 2 → Silhouette = 0.5847
5

Visualização: t-SNE das instâncias

Para visualizar, sub-amostramos ~40 instâncias por dataset e projetamos todas juntas para 2D via t-SNE (com PCA como inicialização). Cada ponto é uma instância individual (um app Android). A cor indica o cluster do dataset ao qual a instância pertence. Os labels com nomes são posicionados no centroide da nuvem de cada dataset no espaço 2D.

Nota sobre features heterogêneas: datasets como defensedroid_apicalls_* têm 4000–6000 features de métricas de grafo, enquanto drebin215 tem 215 permissões estáticas. O PCA independente garante que ambos sejam representados em 20 dimensões antes da comparação, tornando a comparação válida independentemente do tipo de feature original.
// 03 — Visualização

Como ler o gráfico t-SNE

🔵

Pontos individuais = instâncias (apps Android)

Cada ponto representa um aplicativo de um dataset. São ~40 amostras por dataset, escolhidas aleatoriamente. A cor indica o cluster do dataset ao qual o app pertence — não é uma propriedade do app individual.

🏷️

Labels = centroide da nuvem de cada dataset

O nome de cada dataset é posicionado na média (x̄, ȳ) de todas as suas instâncias no espaço 2D. Representa o "ponto central" de comportamento daquele dataset.

🎨

Cor = cluster do dataset (não da instância)

O clustering foi feito no nível de datasets via MMD. Todas as instâncias de drebin215 são azuis porque o dataset todo foi atribuído ao Cluster 0. O t-SNE apenas projeta as instâncias — a cor é herdada do resultado do clustering.

📏

Distância no plano ≠ distância MMD

O t-SNE preserva estrutura local mas distorce distâncias globais. Para comparações quantitativas precisas, use o heatmap de distâncias MMD. O t-SNE serve para inspeção visual de quais instâncias "se misturam" entre datasets.

Principais achados

🔴

stat_androcrawl e stat_mh100k são outliers

Com MMD médio acima de 0.35 para todos os outros datasets e o maior valor do experimento (MMD=0.602 entre si), esses dois datasets capturam comportamentos de malware fundamentalmente diferentes dos demais. androcrawl usa features de tráfego de rede enquanto mh100k usa features estatísticas de larga escala.

🔵

Defensedroid apicalls formam um subgrupo coeso

As três variantes de centralidade de grafo (closeness, degree, katz) têm MMD entre si de apenas 0.030–0.082 — as menores distâncias do experimento. São praticamente intercambiáveis do ponto de vista da distribuição de instâncias.

🟡

Kronodroid emulator ≈ real device

MMD=0.058 entre as duas versões do KronoDroid indica que a distribuição de comportamento de malware é quase idêntica entre emulador e dispositivo real — o que é positivo para validade ecológica de experimentos em emulador.

⚠️

Silhouette=0.58 — estrutura real mas não perfeita

Um score acima de 0.5 indica separação genuína. O fato de k=2 ser ótimo (e não k=3 ou k=4) sugere que a principal variação entre esses datasets é uma única dimensão: features de análise estática/dinâmica vs. features de rede/estatísticas.
